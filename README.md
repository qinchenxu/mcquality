# 装备品质

Minecraft NeoForge 1.21.1 模组。当前版本会为合成产物和战利品中的装备随机分配品质，并按品质倍率调整基础属性。项目的下一阶段会从“单一品质倍率”升级为“稀有度 + 随机词条 + 主动技能 + 详细属性面板”的装备系统。

详细设计文档见 [docs/equipment-system-design.md](docs/equipment-system-design.md)。

## 当前状态

当前已经实现的内容：

- 合成获得的护甲、武器、工具会自动生成品质
- 战利品表产出的护甲、武器、工具会自动生成品质
- 品质数据会写入物品数据组件，并在 tooltip 中显示
- 护甲会影响护甲值、护甲韧性、击退抗性
- 武器和工具会影响攻击力、攻击速度、部分采掘相关属性

当前实现仍然是基础版：

- 每件装备只有一个品质结果
- 属性增益来自固定品质倍率
- 暂无随机词条系统
- 暂无主动技能、被动技能、粒子效果
- 暂无独立的装备详情界面

## 品质档位

| 档位 | 倍率 |
| --- | --- |
| 磨损 | -20% |
| 普通 | +10% |
| 罕见 | +30% |
| 稀有 | +50% |
| 史诗 | +80% |
| 传说 | +100% |

这些档位目前既承担稀有度含义，也直接决定属性倍率。后续重构时，品质会保留为“稀有度层级”，实际强化结果由词条系统决定。

## 当前规则

- 只对护甲、武器、工具一类装备生效
- 品质会在物品生成时写入，不会在普通查看时重复随机
- tooltip 目前只展示摘要信息，不展示详细计算过程
- 品质强化基于原有属性做缩放，而不是凭空添加大量新属性

## 已确认的开发方向

后续版本将按下面的方向迭代：

1. 将“固定品质倍率”升级为“稀有度 + 随机百分比 + 多词条”。
2. 词条按装备类型分池，武器、工具、护甲分别拥有不同词条集合。
3. 高品质装备可获得主动技能词条或更高阶的被动词条。
4. 第一阶段优先支持右键释放技能。
5. 技能会增加匹配的粒子效果。
6. 手持武器时，按 I 键进入物品详细属性面板。
7. 详细面板中会展示装备属性、主动技能、被动技能的说明和精确数值。

## 计划中的系统设计

### 1. 稀有度与词条

后续会把当前单一品质拆成两层：

- 稀有度：决定词条数量、词条品质权重、技能词条出现概率
- 词条实例：决定具体加成数值、显示文本、技能信息和表现风格

这意味着未来装备不再只有“+50%”这样的单一结果，而是可能拥有多条不同类型的属性，例如：

- 攻击伤害提升 12.5%
- 攻击速度提升 4.0%
- 暴击或特殊机制类词条
- 主动技能词条或高级被动词条

### 2. 主动技能

第一阶段的主动技能方案：

- 技能主要挂在高品质装备上
- 由技能词条决定具体技能内容
- 先支持右键释放
- 技能需要服务端判定和冷却控制
- 客户端负责粒子和展示表现

适合作为第一批模板技能的方向：

- 范围斩击
- 护盾类技能
- 冲击或突进类技能

### 3. 技能粒子

技能粒子会作为表现层和技能本体绑定：

- 不同技能拥有匹配的视觉风格
- 粒子效果应与技能实际生效保持一致
- 多人环境下需要保证服务端判定与客户端表现同步

初期更适合按“技能风格”复用粒子，而不是为每个技能都做完全独立的资源。

### 4. I 键详细属性面板

计划中的详细属性面板用于承接完整装备信息，避免 tooltip 过长。面板打开条件为：主手持有受支持武器时按 I 键。

面板预计展示：

- 稀有度
- 词条列表
- 精确百分比和数值
- 主动技能说明
- 被动技能说明
- 触发方式
- 冷却时间
- 粒子或技能风格描述

tooltip 后续只保留摘要，详细信息全部进入独立面板。

## 开发里程碑

### 第一阶段

- 完成稀有度与词条数据结构重构
- 完成随机百分比与多词条生成
- 完成基础主动技能框架
- 完成技能粒子表现
- 完成 I 键详细属性面板

### 第二阶段

- 扩展复杂被动机制
- 增加更多触发方式
- 增加更复杂的战斗状态系统
- 扩展更多 UI 信息组织方式

## 运行

1. 安装 Java 21
2. 在工程目录执行 `gradle genIntellijRuns` 或 `gradle genEclipseRuns`
3. 执行 `gradle runClient`
4. 游戏目录可在 IDE 运行配置里改到 `/Volumes/KINGSTON/hmcl`

如果本机没有安装 Gradle，可以先安装后再生成 wrapper，或直接在 IDE 中导入 Gradle 工程。

## 说明

README 中的“当前状态”描述的是已经落地的实现；“已确认的开发方向”和“计划中的系统设计”描述的是下一阶段准备实现的能力，不代表当前版本已经具备这些功能。

如果需要进入实现阶段，优先参考 [docs/equipment-system-design.md](docs/equipment-system-design.md) 中的数据结构、资源格式、客户端界面与网络同步设计。
